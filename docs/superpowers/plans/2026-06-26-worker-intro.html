What we're building

A Worker is a complete, ready-to-work AI unit. The extension's single strongest capability is turning a raw model into a dependable worker that knows its job, runs reliably, and produces real output. Three parts — and the first three epics prove each one:

Worker = Agent + Harness + Soul Epic 1 proves the Agent · Epic 2 proves the Harness · Epic 3 proves the Soul
Brain
Agent
A Model from a Company, reached over a Protocol (cli/http) via an AuthMethod + Credentials.
Body
Harness
Everything-but-the-model that makes it run reliably: agent loop, tool dispatch, guardrails, observability, memory, verification.
Identity
Soul
The role + responsibilities + methodology that make it a worker: "You are a developer; a developer ⟨investigates → plans → implements → verifies⟩."

Guiding principle — prove the three parts before expanding

"A decent model with a great harness beats a great model with a bad one." So we build narrow and deep first: get one stable agent bridge (Codex CLI), then a real harness on it, then a real soul on it — three small epics that prove Worker works well end to end. Only then do we expand (more CLIs, then HTTP) and add management (CRUD, multi-account). The CLIs ship their own agent loop, so the core is cheap to prove; HTTP — where we build the loop — waits until it's clearly worth the cost.

The epics

Fine-sliced. The first three each prove one part of the Worker on Codex; everything else waits behind them. Each epic still ships something real, and each is chunked further into TDD tasks at plan time.

EPIC 2 Harness Core — make it reliable proves: Harness

Real deliverable: the worker shows exactly what it's doing and refuses to claim "done" without passing checks. The "better harness" value, made visible.

Build the control-system facets on top of Epic 1's bridge: observability (parse the full event stream — tool calls, logs, decisions), a verification gate (lint/test must pass before "done"), and guardrails (sandbox modes, step/token caps, approval). EPIC 3 Soul Core — make it a worker proves: Soul

Real deliverable: pick a role and the Worker visibly behaves like that role — a developer plans-then-verifies, a reviewer critiques. It becomes a real worker, not a raw model.

The identity layer: souls/ library as data (developer, reviewer, qa), render-to-instruction, the verified soul-injection channel (-c override vs temp AGENTS.md — confirmed here), the role picker, and batched decision-requests (§7.2 — enumerate decisions up front, ask once).

Task chunking (TDD, commit per task)

  1. Capture ground truth. Run codex exec --json on a throwaway task, save the real JSONL as a fixture. Build parseEvents against reality, not the (already-wrong) research doc.
  2. Domain types (pure). types.ts + adapters/types.ts — the CLI-agnostic Worker/WorkerEvent/AgentAdapter contracts.
  3. Codex adapter (pure, DI). buildInvocation → codex exec --json -s <sandbox> -C <dir> [-m <model>] "<task>"; parseEvents maps the captured JSONL → WorkerEvents. No process spawned in tests.
  4. Runner. runWorker wires adapter → injected spawn → event stream → cancel handle. Tested with a fake child emitting fixture JSONL.
  5. Host wiring. protocol messages + panel.ts routing + extension.ts command.
  6. Minimal webview. worker.tsx reusing existing shadcn primitives; streams taskEvents into the output pane.

Verified ground truth (codex 0.142.2, checked on this machine)

codex exec+ --json (JSONL stream)+ -s sandbox+ -C dir+ -m model?

Confirmed present: -m, --json, -c key=value, plus (per spec §8) -s, -C, -o, --add-dir, --skip-git-repo-check, --ephemeral. Auth reuses the existing codex login. Epic 1 passes the task as the prompt and does not use -c — soul injection is Epic 3's job, kept out of the bridge.

Out of scope (whole roadmap)

semantic routingscrum-team orchestration — per spec §13. (Cross-session persistence is deferred into Epic 6, not cut.)
