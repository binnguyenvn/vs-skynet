Worker — Epic 1: Codex Adapter

Implementation plan · proves the Agent · single CLI (Codex)

spec: docs/superpowers/specs/2026-06-26-worker-design.md 2026-06-26 TDD · 5 tasks
Worker = Agent + Harness + Soul

Goal: Run any task through the Codex CLI and watch clean, normalized streamed output — and cancel it.

Architecture: A host-side src/worker/ module. A pure AgentAdapter (Codex) turns a Worker into a codex exec --json invocation and parses its JSONL stream into normalized WorkerEvents. runWorker spawns the process, pipes stdout through the parser, streams events to a callback, and returns a cancel handle. The webview gets a thin Worker view; the host owns the runner and credentials.

Verified against the real binary (codex-cli 0.142.2) — not the research doc. Event envelope is a thread/turn/item model: thread.started → turn.started → item.started/completed → turn.completed (terminal). Items: agent_message {text}, command_execution {command, exit_code, status}. Non-JSON noise lines appear — the parser skips them. Spawn with stdio:["ignore","pipe","pipe"] or Codex blocks on stdin.

Global constraints

  • Invocation: codex exec --json -s <mode> -C <dir> [-m <model>] --skip-git-repo-check "<task>"
  • No soul system in Epic 1 — task passed through as the raw prompt; Worker.soul exists but is ignored (Epic 3).
  • Credentials = passthrough — Codex uses its own codex login; no credential store (Epic 6).
  • Type files stay node-free so protocol.ts can import type without pulling child_process into the webview bundle.
  • Tests compile to out/**/*.test.js, run with npm test (Mocha suite/test/assert). Gates: npm run check-types, npm run lint.
