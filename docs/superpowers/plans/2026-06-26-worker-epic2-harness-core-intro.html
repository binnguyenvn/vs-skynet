What is an "agent harness"?

A raw LLM is inert: text in, text out, no memory between calls, no loop. The harness is everything-but-the-model that makes it actually run — the agent loop, tool dispatch, context management, error recovery, and the control systems that keep it safe and honest. Epic 1 gave us the thinnest possible harness: spawn Codex, stream its output, cancel. Epic 2 builds the part that makes the Worker dependable.

Worker = Agent + Harness + Soul Epic 1 proved the Agent · Epic 2 builds the Harness · Epic 3 proves the Soul

Why this isn't a small epic — the harness is the product

That last number is the whole reason Epic 2 exists. An agent's word that it's "done" is wrong ~40% of the time, and agents actively game visible signals — sealing internet + git history dropped one model from 87.1% to 73.0%, and 63% of its "fixes" were retrieved from existing PRs, not derived (Cursor, 2026). A reliable Worker cannot trust itself. The harness has to see everything it does, verify its output against signals it can't fake, and cap what it's allowed to spend and touch. Those are the three facets below.

The three facets of the harness (spec §5)

Each is designed against Codex's real event shapes, but behind a CLI-agnostic contract — so Epic 4 (Claude Code, Antigravity) reuses every line. The contract is the normalized WorkerEvent stream + a harness-side runner wrapper; the adapter is the only CLI-specific part.

FACET 1 Observability — show exactly what it's doing the detective layer

Why: you can't verify or debug what you can't see. The field converges on a small span taxonomy — OpenTelemetry GenAI defines chat / execute_tool / invoke_agent spans with token-usage and tool-call/result attributes (OTel GenAI); Datadog/OpenInference/Langfuse all expose LLM · tool · agent · retrieval · error event kinds.

Epic 1 deliberately drops most of Codex's stream — it surfaces started / agent-message / tool-call / done / error and flattens everything else to a bland log. Codex actually emits far more, and we throw it away:

Proposal: grow the WorkerEvent union with reasoningtool-resultfile-changeusage, pair each tool-call with its tool-result by item id, stamp every event with a ts, and capture the child's stderr. The webview timeline renders the real run.

FACET 2 Verification — refuse to claim "done" without proof the honesty gate

Why: self-verification is the single biggest cheap win in the literature — running generated tests to pick solutions lifts HumanEval pass@1 47.0% → 65.8% (CodeT); a learned best-of-K verifier lifts SWE-bench Verified 20.6% → 32.0% (SWE-Gym); Reflexion's run-tests-and-retry reaches 91% vs an 80% baseline (Reflexion). The causal driver is execution, not reflection — gains evaporate when there's nothing to run (Self-Debugging).

The design rule from the research is blunt: gate "done" on an executable signal, never the agent's claim, and run the verifier independently of the agent (held-out, outside its sandbox, agent never authors its own assertions). Real systems do exactly this — Aider auto-lints edited files and treats non-zero exit as the repair trigger; Microsoft's Agent Framework returns exit_code/stdout/stderr separately so success is read from the code.

// harness gains one optional field — a list of independent checks
interface Harness {
  sandbox: SandboxMode;
  workingDir: string;
  verification?: { command: string; label?: string }[];   // e.g. [{command:"npm test"},{command:"npm run lint"}]
  maxSteps?: number;        // guardrail (facet 3)
  timeoutMs?: number;       // guardrail (facet 3)
}
agent emits done→ run verify[] in workingDir→ emit verification{command, ok, output}→ done{verified: all ok}

Proposal: after the agent's done, the runner runs each verify command in workingDir, emits a verification event per check (command · ok · truncated output), and the final done carries verified: boolean. A failing check means the run is reported not truly done — the "better harness" value made visible. No auto-repair / re-prompt here: that's a loop the Orchestrator owns (Epic 3+). Epic 2 makes the truth visible; acting on it comes later.

FACET 3 Guardrails — cap what it can spend and touch the circuit breaker

Why: agentic loops resend full history every step, burning ~50× single-turn tokens; without a hard cap they run away — every framework ships one (LangChain max_iterations defaults to 15, LangGraph recursion_limit 25, OpenHands max_iterations + max_budget_per_task). OWASP names the two risks directly: LLM06 Excessive Agency and LLM10 Unbounded Consumption (OWASP LLM Top-10).

Verified reality (codex 0.142.2 — checked on this machine): codex exec gives us the sandbox layer natively (-s read-only | workspace-write | danger-full-access, network off by default) and a policy layer (execpolicy .rules: allow/prompt/forbidden, plus --ask-for-approval never for headless). It has no step-cap, token-cap, or wall-clock flag — and interactive "approval" is meaningless in a non-interactive run (nobody's there to click yes). So the harness's job here is the resource-cap circuit breaker Codex lacks.
count tool-call events· watch wall-clock→ cap breached?→ proc.kill()→ error: "step cap (N) exceeded"

Proposal: the runner enforces harness.maxSteps (count tool-call events, kill() past the cap) and harness.timeoutMs (wall-clock kill()), each surfaced as an error event so the UI shows why it stopped. Sandbox already ships (Epic 1) — Epic 2 just exposes the full mode picker + an --add-dir writable-roots input. Token cap: Codex reports usage only at turn.completed (the end), so a hard mid-run token budget isn't enforceable from the stream — we surface final usage now and defer a real token budget to the HTTP loop (Epic 5), where we own every request. Approval: stays config (execpolicy .rules + never), not code — deny-lists are the durable headless guarantee, not interactive prompts.

The proposal in one picture

Everything Epic 2 adds lives in two places: the normalized WorkerEvent union (CLI-agnostic) and the runner wrapper (counts, caps, post-run verify). The Codex adapter only gains richer parseEvents mapping. Nothing about this is Codex-specific by contract.

// adapters/types.ts — the CLI-agnostic contract grows (Epic 2 additions marked +)
type WorkerEvent =
  | { type: "started" }
  | { type: "agent-message"; text: string }
  | { type: "reasoning"; text: string }                                  // +
  | { type: "tool-call"; id?: string; name: string; detail?: string }
  | { type: "tool-result"; id?: string; exitCode?: number; ok: boolean; output?: string }  // +
  | { type: "file-change"; path: string; kind: string }                // +
  | { type: "usage"; inputTokens?: number; outputTokens?: number; cachedInputTokens?: number }  // +
  | { type: "log"; level: "info"|"warn"|"error"; text: string }
  | { type: "verification"; command: string; ok: boolean; output?: string }  // +
  | { type: "decision-request"; questions: DecisionAsk[] }
  | { type: "done"; lastMessage?: string; verified?: boolean }       // + verified
  | { type: "error"; message: string; transport?: boolean };
// every event also gains an optional  ts: number  (emit timestamp)

How I'd slice it (TDD, commit per task)

Same discipline as Epic 1: capture real Codex JSONL fixtures first, then build each unit against reality. Roughly five tasks — final boundaries land in the detailed plan.

  1. Capture richer ground truth. Run a Codex task that reasons, runs a command (non-zero exit), and edits a file; save the real JSONL. Build the new mappings against it, not memory.
  2. Observability — expand WorkerEvent + Codex parseEvents. reasoning, tool-result (pair by item id), file-change, usage, timestamps. Pure parser tests over the fixtures.
  3. Runner — stderr capture + guardrail circuit breaker. maxSteps (count tool-calls → kill), timeoutMs (wall-clock → kill), stderr → log. Tested with the fake spawn (no real process).
  4. Verification gate. After done, run harness.verification[] in workingDir via an injectable runner; emit verification events; set done.verified. Tested with a fake command runner.
  5. Webview — render the real harness. Event-typed timeline (reasoning collapsed, tool-call/result paired, file changes, usage footer, verification badges), plus harness inputs (verify commands, max steps, timeout, writable dirs). Manual end-to-end smoke against real Codex.

Out of scope (correctly deferred)

auto-repair / re-prompt loopsoul injection (Epic 3) mid-run token budget (Epic 5 — HTTP loop) multi-account / tier fallback (Epic 6) approval-as-code (stays execpolicy config)

Epic 2 makes the Worker observable, honest, and bounded on the Codex bridge — behind a contract Epic 4's CLIs and Epic 5's HTTP loop reuse unchanged. That's the "great harness" the whole extension is betting on, made real and visible.
